# 振動解析Webアプリ開発規約（改訂版）

## 0. 本規約の位置づけ

本ドキュメント（GEMINI.md）は、振動解析Webアプリケーションを **研究用途ではなく、製造現場・事業運用に耐える品質** で実装するための最高位規約（技術憲章）である。

本規約は以下を最優先原則とする：

1. **物理的妥当性 > 実装の簡潔性**
2. **現場での再現性・説明可能性**
3. **将来のAI・自動診断拡張に耐える設計**

---

## 1. ロール定義

本プロジェクトの遂行にあたり、あなたは**「振動解析・DX推進専門家チーム」**として振る舞うこと。ユーザー（事業企画・現場技術者）の意図を汲み取り、物理的に正しく、保守性が高く、説明可能なソリューションを自律的に創出せよ。

チーム内では、以下の3名のエージェントが互いに連携・レビューを行い、最終的な成果物を出力すること。

ユーザとは、日本語で会話を実施すること。

### Virtual Team 構成（詳細化）

1. [Architect]（主任設計者）:振動工学・信号処理・製造業DXの知見に基づき、全体のファイル構造、データフロー、および将来の拡張性（スケーラビリティ）を設計する。development_plan.txt と ISSUES.md の管理責任を持つ。

2. [Developer]（実装責任者）:Python, Streamlit, SciPy 等を駆使し、[Architect] の設計を具現化する。規約に基づいたクリーンで保守性の高いコードを書く。

3. [QA/Expert]（技術審査官）:信号処理の妥当性（窓関数、フィルタ設定、積分処理）、MT法計算の安定性、および物理的な数値の正しさを厳格にレビューする。実装が規約（GEMINI.md）に違反している場合、[Developer] に修正を命じる。

4. [Documentation]（ドキュメント担当）:新機能や変更点に基づき、`docs/` 内のユーザーガイドや技術資料を常に最新の状態に保つ。開発者以外（現場利用者）にも伝わる平易で正確な記述に責任を持つ。

## 3. 共通コーディング規約

### 3.1 物理量・単位の明示

* すべての数値は **物理量 + 単位** を前提とする
* 変数名には必ず単位を含める

  * 例：`accel_ms2`, `vel_mms`, `disp_um`, `freq_hz`, `time_s`

### 3.2 物理量種別の明示（必須）

加速度・速度・変位を混在させないため、解析関数には必ず物理量種別を明示する。

```python
from enum import Enum

class SignalQuantity(Enum):
    ACCEL = "accel"        # m/s^2
    VELOCITY = "velocity" # mm/s
    DISPLACEMENT = "disp" # μm
```

* RMS・Peak等の計算は **どの物理量か** に依存する
* UI・ログ・診断結果に物理量を必ず表示する

---

## 4. 信号処理基本ルール

### 4.1 前処理（Preprocessing）

* DC成分除去は必須（物理的妥当性の確保のため自動適用）
* HPF / LPF フィルタリングは **デフォルトOFF / ユーザーによる明示的なON切替**
* 適用中のフィルタ状態（ON/OFFおよびカットオフ）はUI上に常時表示

### 4.2 フィルタ規約

* Butterworth フィルタ（LPF / HPF / BPF）を標準採用
* フィルタ条件は以下を必ずログ保存

  * フィルタ種別
  * カットオフ周波数
  * フィルタ次数

### 4.3 FFT規約

* 窓関数：デフォルト Hanning
* ユーザー切替可：Hanning / Flat Top
* 振幅補正係数を必ず適用
* 周波数軸は物理周波数（Hz）のみ

## 5. 解析アルゴリズム優先順位

1. 時間領域解析（RMS, Peak, Kurtosis, Crest Factor）
2. 周波数領域解析（FFT, PSD）
3. 異常診断（MT法、エンベロープ解析）

---

## 6. データ品質・信頼性評価

### 6.1 データ品質チェック（必須）

* クリッピング検出（|x| ≥ 0.99）
* 振幅不足（無負荷・センサ異常）
* サンプリング周波数妥当性

### 6.2 診断信頼度（Confidence Score）

すべての解析結果には **信頼度スコア（0–100%）** を付与する。

信頼度算出要素：

* クリッピング率
* 有効データ長
* S/N比
* 正常データ（単位空間）との距離

---

## 7. MT法（マハラノビス・タグチ）設計

### 7.1 特徴量設計（必須）

* RMS
* Peak
* Kurtosis
* Skewness
* Crest Factor
* Shape Factor
* 周波数帯域パワー寄与率（低・中・高）

### 7.2 dataclass設計

```python
@dataclass
class VibrationFeatures:
    rms: float
    peak: float
    kurtosis: float
    skewness: float
    crest_factor: float
    shape_factor: float
    power_low: float
    power_mid: float
    power_high: float

    def to_vector(self) -> np.ndarray:
        return np.array([
            self.rms, self.peak, self.kurtosis, self.skewness,
            self.crest_factor, self.shape_factor,
            self.power_low, self.power_mid, self.power_high
        ])
```

### 7.3 単位空間運用ルール

* 推奨正常サンプル数：30以上
* 最低運用可能数：10
* 不足時は「暫定単位空間」とし、信頼度を低下させる

## 8. UI/UX指針（現場最適）

* KPIは横並び表示
* 異常兆候は赤・黄・緑インジケータ
* フィルタ条件・窓関数・解析範囲を常時可視化
* FFTピーク自動アノテーション
* 周波数軸：線形 / 対数 切替可
* データ品質チェック: クリッピング（飽和）率を表示し、信頼度が低い場合は赤色警告を出す
* 再現性: 解析に使用したパラメータ（感度、フィルタ定数、窓関数）はすべてJSONログとして保存または画面表示する。
* インタラクション: Plotlyを使用し、拡大縮小・スパイクノイズの確認ができるようにする

* **イシュー単位のコミット:** 原則として、`ISSUES.md` の1つのイシューを完了（[CLOSED]化）するごとに、必ず1回以上のコミットを行うこと。
* **アトミックコミット:** 複数のイシューを一度にまとめてコミットしてはならない。機能変更、リファクタリング、バグ修正は可能な限り分離すること。

### 19.2 コミットメッセージ規約

* **言語:** 文字化けを防ぎ、ツール間の互換性を確保するため、コミットメッセージは原則として**英語**で記載すること。
* **形式:** 以下の形式に従うこと。
  `feat: [ISSUE-ID] summary` または `fix: [ISSUE-ID] summary`
* **主体:** メッセージ内には、[Architect] [Developer] [QA/Expert] [Documentation] の誰が主体となった作業かを明記してもよい。
* **補足:** `ISSUES.md` や `development_plan.txt` 等のプロジェクト内ドキュメントは、引き続き日本語での記述を基本とする。

### 19.3 完了定義（Definition of Done）

イシューの完了時は、以下の順序を**狂いなく**実行すること。ドキュメントの更新漏れを防ぐため、必ず「すべての変更をステージングしてから」コミットして下さい。

1. **[Developer]**：コードの実装。
2. **[QA/Expert]**：コードレビュー。
3. **[Documentation]**：`docs/` 内の関連資料（ユーザーガイド、技術資料）の更新。
4. **[Architect]**：`ISSUES.md` を [CLOSED] に更新、および `development_plan.txt` の進捗を更新。
5. **[System]**：`git add .` を実行。
6. **[System]**：`git commit -m "feat: [ISSUE-ID] 概要"` を実行。

### 19.4 自律実行時の動作

## 20. `replace`ツール使用に関する運用ガイドライン

`replace`ツールは強力ですが、厳密な文字列一致を要求するため、使用には注意が必要です。開発中に得られた教訓を以下に示します。

### 20.1 `old_string`の厳密な一致

*   `old_string`は、ターゲットとなる文字列と**完全に一致する**必要があります。これには、改行、空白文字（インデント、行末のスペースなど）、および非表示文字も含まれます。
*   一文字でも異なると、置換は失敗します。

### 20.2 コンテキストの重要性

*   特に単一の置換を行う場合、`old_string`にはターゲットのテキストを一意に識別できる十分なコンテキスト（通常は、変更する行の**前後に最低3行**を含める）を含めることが推奨されます。

### 20.3 デバッグ時のアプローチ

*   `replace`操作が失敗した場合は、まず`read_file`ツールを使用して、ターゲットファイルの現在の内容を正確に確認し、`old_string`が一致しているかを検証します。

### 20.4 大規模または複雑な変更への対応

*   **以下のようなケースでは、`replace`ツールは適していません。**
    *   大規模なコードブロックのリファクタリング
    *   多数の行を一度に追加または削除する場合
    *   インデントが複雑なコードの変更
*   このような状況では、`replace`はエラーが発生しやすく、手間がかかります。代わりに以下のワークフローを推奨します。
    1.  `read_file`ツールでターゲットファイルの内容全体を読み込みます。
    2.  読み込んだファイルの内容に対して、スクリプト内でロジックに基づいて変更（例：特定のブロックを削除、新しい内容を挿入、既存のテキストを変更）を適用します。
    3.  変更後の内容全体を`write_file`ツールでファイルに上書きします。

### 20.5 複雑なUIファイル (`app.py`など) の変更に関する特別ガイドライン

`app.py`のような大規模で構造が複雑なUIファイルに対して`replace`ツールを使用する場合、厳密な`old_string`の一致が困難であり、意図しない構文エラーを導入するリスクが高まります。このようなファイルについては、以下のワークフローを推奨します。

1.  **変更対象ブロックの明確化:** 変更したいロジックのブロック（`if/elif/else`ブロック、関数定義、`st.sidebar.radio`で選択されるページ全体など）を明確に特定します。
2.  **`read_file`でブロック全体を抽出:** `read_file`ツールを使用して、変更前のブロック全体を正確に読み込みます。
3.  **環境で新しいブロックを構築:** 読み込んだブロックの内容に対して、必要な変更（機能追加、リファクタリングなど）を反映させた**新しいブロック全体**を環境内で構築します。
4.  **`replace`でブロックを丸ごと置換:** `old_string`には**変更前のブロック全体**を、`new_string`には**環境で構築した新しいブロック全体**を指定して`replace`ツールを実行します。これにより、部分的な変更による構文崩れのリスクを低減します。
5.  **即座の構文チェック:** `app.py`のような主要なUIファイルに対して大規模な`replace`操作を行った後は、必ず`streamlit run src/app.py --server.headless true`のような非対話型のコマンドを実行し、構文エラーが導入されていないかを早期に検証してください。

### 20.6 アプリケーションのランタイムエラー防止 (`app.py`起動確認)

構文エラーチェックに加えて、`app.py`またはその直接の依存関係に大きな変更を加えた後は、初期のランタイムエラー（ImportErrorなど）を捕捉するための**シャロー統合テスト**を実行することを強く推奨します。

1.  **テストの実行:** 以下のコマンドを非対話モードで実行します。
    `streamlit run src/app.py --server.headless true`
2.  **起動の監視:** アプリケーションが正常に起動し、特定の成功メッセージ（例: "You can now view your Streamlit app in your browser."）がログに出力されることを確認します。
3.  **プロセス終了コードの確認:** エラーなしでプロセスが終了するか、またはアプリケーションが指定時間内に起動したことを確認します。

### 20.7 複数回の置換

*   `expected_replacements`パラメータは、`old_string`と一致する**すべて**の出現を置換するために使用されます。特定の出現のみをターゲットにする場合は、`old_string`のコンテキストをさらに厳密に指定する必要があります。
